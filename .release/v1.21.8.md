# Release Notes - v1.21.8

**Release Date:** April 3, 2026

## Overview

This release improves tool calling functionality across the SDK and BUI by fixing how tool function names are injected into cached message contexts. The changes ensure that tool responses render correctly for both standard templates (like Gemma 4) and GPT-style templates. Additionally, the examples have been updated to use catalog-based model IDs instead of direct HuggingFace URLs, and the agent tool calling flow has been enhanced with better output formatting and error handling.

## Bug Fixes

### Tool Response Name Injection (#445)

Fixed a critical issue where tool function names were not being properly injected into cached messages, causing tool calls to fail in subsequent requests when using message caching (SPC/IMC):

- **Unified name injection**: Renamed `gptInjectToolCallNames` to `injectToolResponseNames` and removed the GPT-model-only restriction. Tool names are now injected for all model templates, not just GPT-style ones.

- **Dual template support**: The function now injects both `name` (used by standard templates like Gemma 4) and `tool_call_name` (used by GPT templates) fields into tool response messages, ensuring compatibility across different model architectures.

- **Smart injection logic**: Added checks to avoid overwriting existing name values. If both `name` and `tool_call_name` are already present on a message, the function skips that message entirely. Each field is injected independently if missing.

- **Improved logging**: Updated log messages to reflect the broader scope of the injection logic and removed GPT-specific terminology.

## Examples Updates

### Agent Tool Calling Enhancements

Improved the agent example's tool calling flow with better user feedback:

- **Unknown tool detection**: Added visual indication when an unknown tool is called, displaying "Unknown tool: {name}" in red text.

- **Tool call status display**: Changed tool output format to show `toolName(args): ok` or `toolName(args): "FAILED"` on a single line instead of multiple lines, making the output more concise.

- **Failed tool detection**: Added logic to detect and display failed tool responses by checking if the content contains `"FAILED"`.

### Catalog-Based Model Configuration

Updated model configuration in examples to use catalog model IDs instead of direct HuggingFace URLs:

- Changed `SourceURL` to be commented out
- Enabled `ModelID: "gpt-oss-20b-Q8_0"` for catalog-based model resolution

### Response Formatting Improvements

Fixed reasoning output formatting and latency printer behavior:

- Removed redundant newline injection during first reasoning chunk
- Added proper spacing after stopping the latency printer on first real chunk
- Simplified variable naming from `reasonFirstChunk` to `firstChunk`

## Technical Details

### Tool Response Name Injection Flow

The `injectToolResponseNames` function follows this flow:

1. Build a mapping from `tool_call_id` to function name by scanning assistant messages with tool_calls
2. Iterate through all messages looking for role:"tool" messages
3. For each tool message, check if it has a matching `tool_call_id` in the map
4. Check existing name fields to avoid overwriting
5. If either `name` or `tool_call_name` is missing, inject the function name using copy-on-write semantics

### GoCodeEditor Line Number Handling

The `GoCodeEditor.Call` function now handles line number arguments as both `float64` (from JSON) and `string` types, with proper type switching and error handling:

```go
var lineNumber int
switch v := toolCall.Function.Arguments["line_number"].(type) {
case float64:
    lineNumber = int(v)
case string:
    lineNumber, _ = strconv.Atoi(v)
}
```

## Migration Notes

No breaking changes for existing clients. However, developers working with tool calling should be aware:

1. **Tool name injection is now universal**: The tool response name injection logic applies to all model types, not just GPT models. This improves compatibility but may affect caching behavior for models that don't require name injection.

2. **Dual field support**: Tool response messages now include both `name` and `tool_call_name` fields when applicable. Templates should check for either field.

3. **Example updates**: The agent example has been updated to use catalog model IDs. Users following the examples should ensure their catalog is up to date or adjust the configuration accordingly.
